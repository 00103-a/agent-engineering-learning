## 已验证的代码练习

- 使用两次 `await` 串行查询天气和活动，总等待时间由两段耗时相加。
- 使用 `asyncio.gather()` 并发查询并正确解包为 `weather` 和 `activities`。
- 观察到任务按实际耗时完成，但 `gather()` 结果仍按输入顺序排列。
- 使用 `asyncio.wait_for()` 为天气查询设置短于执行耗时的超时边界。
- 使用 `try/except asyncio.TimeoutError` 将异常转换为 `(False, None)` 固定结果。
- 使用 `finally` 验证协程超时取消后仍会执行资源清理路径。
- 使用 `asyncio.as_completed()` 按完成顺序收集活动和天气结果。
- 使用共享的 `asyncio.Semaphore(2)` 限制四个任务的同时执行数量，并验证槽位动态释放。
- 使用 `if __name__ == "__main__"` 避免模块被测试导入时自动执行入口代码。

## 今日关键认识

- 调用 `async def` 产生 Coroutine；当前需要结果时使用 `await`，从同步入口启动时使用 `asyncio.run()`。
- `gather()` 适合全部完成后按输入位置组合结果；`as_completed()` 适合完成一个就处理一个。
- `as_completed()` 不读取函数体预测耗时，而是调度任务并观察实际完成事件。
- `Semaphore` 限制同一时刻的执行数量，不固定任务批次，也不限制每秒请求数。
- 超时取消正常执行路径，但 `finally` 仍应完成本地资源清理。

## Agent 知识输入

### 多工具 Runtime

LLM 提出工具调用，Runtime 负责注册表查找、参数验证、依赖判断、并发策略、超时、重试、身份对应和结果记录。并发工具结果需要保留 `tool_call_id`、状态和结构化结果，不能只保存一个无法对应来源的混合列表。

### Agent 与 Workflow

Workflow 的路线由程序预先确定；Agent 在系统边界内根据 Observation 动态选择下一步。生产系统通常使用确定性外层流程包围 Agent 探索阶段，高风险副作用仍由程序规则和人工确认控制。

### 上下文工程

记忆系统负责保存信息，上下文工程负责选择本轮模型真正需要的信息。上下文应区分稳定规则、当前任务、Run 状态和外部知识，并处理来源、时效、冲突、压缩和 Token 成本。

## 暴露的薄弱点

- 初次编写时遗漏 `await asyncio.sleep(...)`，运行后通过 `RuntimeWarning` 定位。
- 曾把函数对象直接传给 `gather()` 或 `await`，需要继续稳定“函数调用后才产生协程对象”的判断。
- 忘记同步程序使用 `asyncio.run()` 启动最外层协程。
- 多行函数参数曾遗漏逗号，基础 Python 语法仍需在后续新任务中自然复现。
- `as_completed()` 的内部等待方式需要多次解释，当前只要求理解用途。
