# Agent 工程学习设计文档

## 1. 文档目的

本文件用于固定 Agent 工程学习的目标、范围、优先级、教学方法和成果标准，避免学习过程被热门框架、零散教程或过度工程化带偏。

后续课程、源码阅读、独立练习和知识库整理均以本文件为基准。路线可以根据岗位变化和学习反馈调整，但每次调整都需要说明原因。

## 2. 学习者背景

- 2026 年下半年进入大三阶段。
- 目标是准备 2027 年寒假和暑假实习。
- 求职方向包括 Agent 工程师、LLM 应用工程师、AI 后端工程师，以及全栈开发岗位。
- 当前阶段优先学习 Agent，后续逐渐推进全栈能力。
- 每天可稳定投入 3～4 小时，其中适合高强度编码和推演的时间为 1～2 小时。
- 练习代码使用独立项目，不读取、不修改 DayAgent。

### 2.1 课前诊断结论（2026-07-15）

课前诊断的 7 个测试最终全部通过，但 Mini Agent Runtime 是在逐步提示下完成的，因此只能证明学习者能够跟随解释理解和修改代码，不能视为已经具备独立实现能力。

当前已表现出的优势：

- 能理解 Workflow 与 Agent 的控制权差异。
- 能在具体例子中理解 Run、Step、Tool Call、终止条件和错误恢复。
- 愿意追问代码为什么这样运行，而不是只复制最终答案。
- 能在分解后完成工具注册、有限重试和 Agent Loop 的完整闭环。

当前主要缺口：

- Python 字典、数据类、类型标注和函数契约尚未形成熟练使用能力。
- 对异常对象、控制流和错误分类的理解需要通过小练习巩固。
- `async`、`await`、超时、并发和任务生命周期仍处于入门阶段。
- 能跟随局部提示实现，但暂时无法根据测试和接口独立完成综合题。

因此，近期不直接进入复杂 Agent Runtime 源码。先进行 Agent 场景驱动的 Python 工程基础训练，再逐步回到 Runtime 和框架源码。

## 3. 总体定位

学习目标不是熟练背诵某个 Agent 框架，而是具备设计、实现、测试和交付 Agent 应用的工程能力。

目标能力模型采用 T 型结构：

```text
深度方向：Agent 机制 + Python 后端工程
广度方向：前端交互 + 数据库 + 部署与运维基础
```

面向 2027 年实习，主要岗位定位为：

- AI Agent 应用工程师
- LLM 应用工程师
- AI 后端工程师
- 全栈 AI 工程师
- Python 后端或全栈开发工程师

暂时不以大模型训练、推理引擎和大型 Agent 平台基础设施岗位为主。

## 4. 核心原则

### 4.1 学问题，不押注框架

框架 API、装饰器和配置语法可能快速变化，但以下问题长期存在：

- 工具参数如何验证。
- 调用失败后是否重试。
- 重复执行是否产生副作用。
- 长任务如何保存和恢复。
- 上下文如何选择和压缩。
- 记忆如何更新、过期和纠错。
- Agent 行为如何追踪、回放和评估。
- 高风险动作如何授权和审计。

每学习一个框架功能，都要回答：

1. 它解决了什么生产问题？
2. 它对应的框架无关抽象是什么？
3. 换模型或框架后，哪些部分仍然保留？
4. 不依赖该框架时，最小实现是什么？
5. 如何测试它确实解决了问题？

### 4.2 自建 Runtime 是学习手段

手写最小 Agent Loop 用于理解框架隐藏的生命周期、状态和错误处理，不以重新发明完整框架为目标，也不把自研框架作为主要简历卖点。

Runtime 学习采用三级递进：

1. 看懂并修改已有的局部实现。
2. 根据接口、测试和分级提示补全实现。
3. 不看旧实现，根据需求和测试独立重写并解释取舍。

三级递进不要求连续重做同一道题。完成逐步提示后，先在后续新任务中螺旋复用旧能力；只有学习者已积累连续成功体验、且相同机制在不同场景中出现过，才安排第三级阶段验收。只有达到第二级后才开始精读框架 Runtime 源码；第三级作为进入复杂编排、上下文和记忆模块前的长期阶段验收。

### 4.3 能确定的部分交给程序

确定性流程优先使用普通代码和 Workflow。只有执行路径需要根据模型判断和环境反馈动态变化时，才使用 Agent。

### 4.4 工程验证优先于功能堆叠

一个有工具契约、失败测试、Trace 和评估集的单 Agent，优先级高于一个缺乏验证的多 Agent Demo。

## 5. Agent 学习的六个层面

### 5.1 Agent 核心运行机制

学习深度：深入实现。

内容包括：

- 模型消息与结构化输出
- Function Calling
- Agent Loop
- 多步骤状态传递
- ReAct 与 Plan-and-Execute
- 并行工具调用
- 终止条件和最大步数
- 流式事件
- 时间、Token 和费用预算
- 中断与取消

前置要求：能够独立使用字典、数据类、异常、异步函数和 pytest 完成单职责小部件。

完成标准：能够脱离框架实现一个小型 Tool Calling Runtime，使用假模型和假工具完成确定性测试，并能解释框架替代了哪些工作。第一次在逐行提示下通过测试只算学习记录，不算完成该标准。

### 5.2 工具契约与可靠性

学习深度：深入实现，并贯穿全程。

内容包括：

- 类型化输入输出
- JSON Schema
- 参数验证
- 错误分类
- 超时传播
- 有限重试
- 幂等性
- 只读与副作用工具
- 权限审批
- 并发调用
- 工具结果大小控制
- Fake Tool 与集成测试

工具设计不只关注如何调用，还要说明副作用、权限要求、超时策略、重试条件和幂等行为。

### 5.3 上下文、RAG 与记忆

学习深度：深入设计和评估。

上下文部分：

- 信息选择与排序
- Token 预算
- 历史压缩
- 来源与可信度
- 不可信内容隔离
- Prompt Injection 边界

RAG 部分：

- 文档解析与 Chunk
- Embedding 与向量检索
- 关键词和混合检索
- Rerank
- 引用与来源
- Recall、Precision、MRR 等基础指标
- 检索是否改善最终任务结果

记忆部分：

- 工作记忆与长期记忆
- Episodic、Semantic、Procedural Memory
- 原始事件、派生画像和检索索引
- 写入策略
- 更新、过期、冲突与删除
- 记忆污染
- 记忆带来的实际效果评估

记忆学习排在基本 Loop 和工具系统之后。没有评估的记忆系统只能算更复杂的存储。

### 5.4 可观测、评估与安全

学习深度：深入实践。

可观测部分：

- `run_id`、`step_id`、`tool_call_id`
- 模型和工具耗时
- Token 与费用
- 错误分类
- 状态变化
- 结构化事件
- 执行轨迹回放
- OpenTelemetry 基础

评估部分：

- 固定测试集
- 最终结果评估
- 工具选择与参数评估
- Agent 轨迹评估
- LLM-as-Judge 的使用与局限
- 回归测试
- Prompt、模型和架构的对照实验

安全部分：

- Prompt Injection
- 工具权限
- Secret 隔离
- 人工确认
- 数据泄露
- 最小权限原则
- 审计事件

复杂容器沙箱可以后置，但参数验证、副作用标记、权限意识和审计入口必须从早期设计开始存在。

### 5.5 主流框架与标准协议

学习深度：一个主框架深入，其他框架对比。

不提前把某个框架固定为长期主框架。进入本阶段时，根据版本活跃度、岗位需求、文档质量和前面已掌握的抽象重新选择一个主框架。LangGraph 是当前候选之一，可用于观察：

- Typed State
- Node 与 Edge
- 条件路由
- Checkpoint
- Interrupt 与 Resume
- Human-in-the-loop
- 失败恢复

OpenAI Agents SDK 用于理解：

- Runner
- Session
- Guardrail
- Handoff
- Tracing

Pydantic AI 用于理解：

- 类型安全
- 结构化结果
- 依赖注入
- Test Model
- Evals

MCP 作为当前主流互操作协议，需要完成一个小型 Server 和 Client，理解 Tool、Resource、传输、认证与权限边界。MCP 的具体版本和字段不作为永久知识。

A2A 和 ACP 当前只要求理解用途，出现真实跨 Agent 或跨宿主边界时再深入。

框架学习的验收对象不是 API 记忆，而是以下可迁移问题：运行循环、类型化状态、条件路由、持久化、暂停恢复、工具审批、执行轨迹和错误边界。主框架若被替代，只需用新框架重新映射这些问题，不重走基础阶段。

### 5.6 够用且扎实的生产工程

学习深度：支撑 Agent 产品独立运行和求职展示。

需要掌握：

- Python 类型、异步和异常处理
- FastAPI
- PostgreSQL 和基础 SQL
- Redis 的缓存或任务用途
- pytest
- Docker
- Git
- 配置与 Secret 管理
- 基础 CI
- HTTP、SSE 与 WebSocket
- 基础并发与进程模型

暂时不深入：

- Kubernetes
- 分布式 Lease
- 大规模事件总线
- 多租户 Agent 平台
- GPU 推理部署
- 服务网格
- 复杂云原生体系

## 7. 明确排除的内容

当前实习准备主线不包含：

- 大模型预训练
- RLHF 与 Agentic RL
- 复杂模型微调
- CUDA 和推理引擎
- 角色扮演式多 Agent Demo
- 大型分布式 Agent 平台
- 完整 Kubernetes Sandbox
- 同时精通多套 Agent 框架
- 低代码平台操作
- 为了使用 Redis、MCP 或向量数据库而强行使用

多 Agent 只在单 Agent、工具、状态和评估能力扎实后再进入。

## 8. Agent 技术的保值策略

未来可能快速贬值：

- 某个框架的装饰器和配置语法
- 手工拼接 ReAct Prompt
- 特定模型的消息字段
- 简单 Planner/Executor 套壳
- 角色 Prompt 驱动的多 Agent
- 某个向量数据库的基础 CRUD

长期更稳定：

- Python、异步、HTTP、SQL、Docker 和测试
- 工具 Schema、参数验证、超时和幂等
- 状态机、Checkpoint、中断恢复
- 上下文选择、压缩和来源管理
- 记忆生命周期和效果评估
- Trace、回放、评估和权限
- 模型与工具之间的标准化接口

学习任何技术时，都要同时掌握“当前实现”和“背后的稳定抽象”。这样即使框架被替换，也只是迁移工具，而不是重新学习职业。

## 9. 后续全栈学习的价值

全栈学习不是偏离 Agent，而是让 Agent 能力形成真实产品。

主要收益：

1. 扩展求职范围，可同时覆盖 Agent、AI 后端、Python 后端和全栈岗位。
2. 能将命令行或 Notebook Demo 变成可运行、可体验的应用。
3. 能处理 SSE/WebSocket、工具事件、审批、取消、恢复和 Trace 展示。
4. 能从浏览器、HTTP、Agent Runtime、Tool、Database 到外部 API 完整调试。
5. 能理解用户如何观察、控制和纠正 Agent，提高产品判断力。
6. 即使 Agent Loop 被模型平台内置，仍能完成业务系统、数据、权限和前端集成。

全栈能力采用 T 型策略：

```text
主深度：Agent + Python 后端
辅助广度：前端 + 数据库 + 部署
```

长期建议投入比例：

```text
Agent 原理与实现：60%
Python 后端和测试：30%
前端与部署：10%
```

当前两周投入比例调整为：

```text
Python-for-Agent 实现与测试：60%
Agent 机制理解与执行推演：30%
笔记整理和轻量源码对照：10%
```

每天安排 1～2 小时核心深度练习，结构为：

```text
10～15 分钟：复述与预测
30～60 分钟：一个单职责编码任务
15～30 分钟：运行测试、定位失败
10～15 分钟：用自己的话解释控制流
```

其余可用时间用于复习、整理、阅读或休息，不强制继续高强度编码。

Agent 核心能力形成体系后，可逐渐调整为：

```text
Agent：40%
后端工程：35%
前端产品化：25%
```

如果已有 Vue 或 Java 经验，不需要为了追逐岗位关键词立即更换技术栈。独立 Agent 练习以 Python 为主，完整产品可以复用已有前端和后端经验。

## 11. 教学与参与方式

每个主题采用交互式工作坊：

```text
定位问题
→ 展示少量源码
→ 学习者先预测
→ 跟踪真实执行
→ 学习者补最小实现
→ 测试成功与失败路径
→ 对比仓库实现
→ 进行工程批判
→ 面试式复述
```

教学约束：

- 不以长篇纯文字源码总结作为主要形式。
- 不在学习者尝试前直接给出完整答案。
- 卡住时优先给最小提示。
- 每次只引入一个主要的新机制，避免同时考察字典、异常、异步和 Agent Loop。
- 提示分为概念提示、结构提示、局部代码和完整示例四级，并记录完成时所需提示等级。
- 学习者连续卡住时先退回更小的 Python 示例，再回到 Agent 代码，不把语法困难误判为 Agent 概念困难。
- 采用螺旋式复现：新任务约 70% 是新目标，约 30% 自然复用已学机制，不要求刚完成提示学习就从空白重写原题。
- 优先通过新场景中的近迁移检验掌握程度；完整空白重写只用于阶段验收，不作为日常惩罚式重复。
- 保持可见的成功反馈。连续出现 C/D 级提示时先缩小任务，积累一次独立通过后再增加难度。
- 每日采用双轨结构：45～70 分钟编码主练习只引入一个核心机制；精力有余时增加 20～40 分钟 Agent 知识输入，不要求当日编码。
- 纯听讲内容只标记为“已接触”，不能计入独立掌握；后续遇到相关实现时再通过推演、编码和测试完成转化。
- 每课只安排一个主要编码任务。
- 必须包含成功路径和失败路径测试。
- 必须区分源码事实、分析判断和未来预测。
- 默认不加入 DayAgent 映射。

一个主题只有在学习者能够定位源码、推演执行、在约定提示等级内实现、完成测试并解释取舍后，才算真正完成。跟随逐行提示完成的代码不能直接标记为掌握，应在后续新任务中复现同一机制；阶段成熟后再安排完整独立实现。

## 12. 求职前的成果标准

到投递 2027 年实习时，应能够：

1. 手写并解释最小 Agent Loop。
2. 设计可靠的工具契约和错误恢复。
3. 使用至少一种届时主流的编排框架实现有状态、可恢复、可审批的流程，并能说明其框架无关抽象。
4. 构建并评估上下文、RAG 和记忆系统。
5. 保存并回放 Agent Trace。
6. 使用 FastAPI、数据库、测试和 Docker 交付完整服务。
7. 实现一个 MCP Server 和 Client。
8. 对架构选择说明为什么使用、替代方案是什么、生产版本还缺什么。
9. 形成至少一个可运行、可测试、可展示的完整 Agent 项目。

## 13. 当前边界结论

本阶段以 Agent 应用工程为核心，以 Python 后端能力作为承载；深入运行机制、工具可靠性、上下文、记忆、评估和权限，保持对主流框架与协议的实践能力，但不进入模型训练和大型 Agent 平台基础设施。

全栈学习在 Agent 核心能力形成后逐步增加，用于产品化、扩大岗位范围和提升端到端交付能力。

当前主线明确为：

```text
Agent 工程
→ 当前训练重点：Python-for-Agent
→ 阶段验收：根据接口和测试独立实现最小 Runtime
→ 验收后进入主流框架源码、上下文、记忆、评估与产品化
```

当前阶段不是暂停 Agent 学习，而是使用 Agent 的真实组件作为 Python 工程训练载体。

## 14. 文档维护机制

本设计文档是持续更新的活文档，可以根据每日学习结果、源码变化、岗位需求和阶段反馈进行调整。

每日学习结束后按需更新以下内容：

- 当日已验证的新结论
- 原有判断的修正及原因
- 学习范围或优先级变化
- 新发现的技术风险和版本变化
- 已完成的能力与仍然薄弱的部分
- 下一阶段的学习决策

更新时遵守以下规则：

1. 不因单个案例或一篇文章立即改变总体路线。
2. 重要调整必须记录日期、依据和影响范围。
3. 区分源码事实、实验结果、个人判断和未来预测。
4. 已过时的内容应标记或修订，不保留相互冲突但无解释的结论。
5. 总设计文档只保留稳定方向，具体课程细节进入独立课次文件。
6. 每完成一个阶段进行一次整体复盘，而不是每天大幅改动目标。

设计文档维护记录统一写入 `CHANGELOG.md`。
