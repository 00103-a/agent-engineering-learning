# Python-for-Agent 两周学习计划

## 计划定位

执行周期：2026-07-15 至 2026-07-28。

本阶段不追求继续增加 Agent 名词，而是把已经接触过的 Registry、Tool Executor、Run、Step 和 Fake Model 拆成单职责练习，补齐实现它们所需的 Python 工程基础。

每天核心深度学习 1～2 小时。完成核心任务后即可停止；额外时间只用于复习、整理或轻量阅读。

## 完成等级

每次练习记录实际使用的提示等级：

- A：独立完成，只查标准库或错误信息。
- B：使用概念提示后完成。
- C：使用控制流或数据结构提示后完成。
- D：需要局部代码或逐行带写。

D 级完成代表已经理解当次讲解，但不代表掌握；相同机制需要在后续用 A 或 B 级再完成一次。

## 每日节奏

```text
10～15 分钟：复述上一课，先预测代码行为
30～60 分钟：完成一个单职责编码任务
15～30 分钟：运行测试，阅读第一个失败并修正
10～15 分钟：口头或文字解释输入、状态、输出和失败
```

卡住 10～15 分钟后可以提问。提问时说明题目、当前理解、尝试和具体卡点，由导师按 B、C、D 顺序逐级提示。

## 第一周：Python 工程部件

### Day 1：字典与工具注册表

- 从空文件实现名称到 `ToolSpec` 的映射。
- 练习字典写入、按键读取、成员判断和 `KeyError`。
- 用测试覆盖成功注册、空名称、重复名称和未知名称。
- 产出：先能解释注册、查找和异常路径；之后在新的工具场景中再次使用并补全 `ToolRegistry`，不单独返工原题。

### Day 2：数据类、类型与函数契约

- 理解 `ToolCall`、`ToolSpec`、`ToolExecution` 各自表示的数据。
- 区分函数对象、调用参数和函数返回值。
- 练习 `str | None`、`dict[str, Any]`、`Callable` 和 `Awaitable` 的阅读。
- 产出：根据字段说明创建数据类，并写三个构造与访问测试。

### Day 3：异常与错误分类

- 练习 `raise`、`try/except`、异常对象和 `str(error)`。
- 区分 transient、permanent、timeout 和 unexpected。
- 判断哪些错误可由代码重试，哪些应交给上层处理。
- 产出：同步版安全执行器及错误分类测试。

### Day 4：`async`、`await` 与执行顺序

- 区分协程对象和已经执行的结果。
- 推演串行 `await` 与 `asyncio.gather` 的执行顺序。
- 观察异常传播和事件循环结束时的任务状态。
- 产出：三个小型异步实验和预测记录。

### Day 5：超时与有限重试

- 使用 `asyncio.wait_for` 设置超时。
- 明确 `max_retries` 与总尝试次数的关系。
- 练习成功、暂时失败后成功、永久失败和最终超时。
- 产出：独立实现一个小型异步 `execute_tool`。

### Day 6：pytest 与测试替身

- 学会从测试名称、输入和断言反推函数契约。
- 使用 `pytest.raises` 验证异常。
- 理解 Fake Tool 与 Scripted Model 为什么能提供确定性。
- 产出：先写失败测试，再完成最小实现使其通过。

### Day 7：螺旋复习与第一次迁移整合

- 使用一个新的工具场景完成小型整合，其中自然包含 Registry 查找、参数契约和 Tool Executor 调用。
- 新任务约 70% 聚焦当前主题，约 30% 复用旧机制；允许查看个人笔记和接口，不直接复制旧函数实现。
- 记录提示等级、首个失败、修复原因和仍不确定的问题。
- 未达到 B 级时缩小下一次任务，并在后续课程再次嵌入同一机制，不立即安排整题重写。

## 第二周：回到 Agent Runtime

### Day 8：Run、Step 与状态轨迹

- 用具体任务画出 Run 和 Step 的变化。
- 区分模型回复、工具调用指令、工具配置和工具执行结果。
- 产出：手工推演两步 Agent 轨迹，并创建对应数据对象。

### Day 9：循环与终止条件

- 实现最大步数循环。
- 区分成功返回、正常达到上限和异常退出。
- 理解为什么最终回答所在步骤也要进入轨迹。
- 产出：只支持 Fake Model 和 final answer 的最小循环。

### Day 10：工具选择与未知工具

- 遍历 `reply.tool_calls`。
- 通过名称从 Registry 获取 `ToolSpec`。
- 将未知工具记录为当前 Step 错误，不让整个 Run 崩溃。
- 产出：未知工具后下一轮能够恢复的测试。

### Day 11：Observation 与错误反馈

- 使用 Tool Executor 执行已注册工具。
- 将成功结果写入 `observations`，失败写入 `errors`。
- 确认下一轮模型只能依据显式记录继续推理。
- 产出：工具调用后返回最终答案的两步测试。

### Day 12：确定性 Runtime 测试

- 覆盖成功、未知工具、永久失败、超时和最大步数。
- 学会一次只读取并解释第一个失败。
- 产出：可重复运行的 Runtime 测试集。

### Day 13：新场景中的 Runtime 迁移

- 使用不同于天气示例的新任务，将状态、工具调用、错误记录和终止条件组合起来。
- 只对尚未学过的部分提供结构提示，旧机制通过新场景自然复现。
- 产出：在 A 或 B 级完成一个 Runtime 相关闭环；不要求从空白复制整套旧 Runtime。

### Day 14：阶段验收与路线决策

- 用自己的话解释 Registry、ToolCall、ToolSpec、ToolExecution、Step 和 RunResult。
- 推演一次成功和一次失败路径。
- 检查测试结果和提示等级，而不是只看代码是否全绿。
- 达到 B 级：先复核届时活跃框架，再进入一个最小调用链的源码学习；`smolagents` 是当前首个候选，不是永久绑定。
- 仍需 C 或 D 级：针对薄弱机制补练，不提前叠加框架复杂度。

## 阶段通过标准

满足以下条件后进入 Agent Runtime 框架源码：

1. 能根据类型标注说明主要数据对象的键、值和职责。
2. 能独立写出字典注册、读取和异常分支。
3. 能解释协程对象、`await`、超时和有限重试。
4. 能从 pytest 断言反推函数契约，并定位第一个失败。
5. 能在 A 或 B 级完成一个新的 Runtime 相关闭环，并在后续阶段完成整套最小 Runtime 的独立实现。

未通过时只补具体薄弱项，不重新学习全部内容，也不把一次失败解释为不适合 Agent 工程。
