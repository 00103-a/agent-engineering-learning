# 下一课学习上下文

> 更新时间：2026-07-21。此文件是下一次教学的首要入口；开始课程前先读取本文件、`00-学习设计文档.md`、`01-两周学习计划.md` 和最近一篇学习记录。

## 1. 学习目标与边界

- 求职方向：Agent 工程师或全栈工程师，重点准备 2027 年寒暑假实习。
- 当前主线：Agent 工程，使用 Python 工程能力承载 Runtime、工具执行、上下文和记忆等主题。
- 每天可投入 3～4 小时，其中适合深度编码的时间为 1～2 小时。
- 当前不读取或改造 DayAgent 代码，也不默认讨论对 DayAgent 的启发。
- 不追逐单一框架 API；优先掌握工具契约、执行状态、超时、并发、错误、预算和测试等稳定抽象。

## 2. 教学方式

- 每日双轨：45～70 分钟编码主练习；精力有余时增加 20～40 分钟 Agent 知识输入。
- 每次编码只引入一个主要新机制，新任务约 70% 新内容、30% 复用旧能力。
- 不立即重写刚完成的原题。通过不同场景中的近迁移复现旧机制，完整重写留作后续阶段验收。
- 学习者先写；卡住后依次提供概念提示、结构提示、局部代码，最后才给完整示例。
- 纯听讲内容只标记为“已接触”，不能算作会实现或独立掌握。
- 以实际完成状态推进，不根据日期或 `Day N` 自动跳课。

## 4. 已接触但未验收的 Agent 知识

- 多工具 Runtime：模型提出调用，Runtime 负责验证、依赖、并发、超时、重试、身份和结果记录。
- Agent 与 Workflow：路径固定时由程序编排；路径需根据 Observation 动态变化时才让 Agent 决策。
- 上下文工程：记忆负责保存信息，上下文工程负责为当前模型调用选择信息。

以上内容下一课不得直接当作学习者已经会设计或实现。

## 5. 下一课核心主题

**并发工具结果的身份对应与部分失败。**

选择这个主题的原因：它把今天的异步基础接回 Agent Runtime，同时只新增一个核心问题：多个工具并发后，如何知道每个结果属于哪个 `tool_call_id`，以及单个失败时如何保留其他成功结果。

### 新旧比例

- 新内容约 70%：调用身份、结构化批量结果、部分失败语义。
- 旧内容约 30%：`gather()`、超时、异常捕获和简单 pytest 断言。
- `Semaphore` 只在主流程完成后作为可选扩展，不与身份建模同时教学。
- 暂不要求使用 `as_completed()`，避免同时处理完成顺序与身份映射两项难点。

## 6. 下一课编码产出

新建独立场景，不修改 2026-07-21 的练习：

```text
exercises/2026-07-22-tool-batch/
├── tool_batch.py
└── test_tool_batch.py
```

目标闭环：

1. 两个带唯一 `call_id` 的工具请求并发执行。
2. 每个结果显式保存 `call_id`、`status`、`value` 和 `error`。
3. 一个工具失败时，另一个成功结果仍被保留。
4. `gather()` 的位置映射与 `call_id` 双重对应能够被测试验证。
5. 学习者至少独立完成结果构造、一次异常分支和两个断言。

不把完整 Registry、重试器和 Agent Loop 一次性塞入本课。

## 7. 下一课开场

先给出以下场景，不直接给代码：

```text
call-weather 返回 "晴天"
call-calendar 抛出临时错误
```

开场问题：如果批量执行只返回 `["晴天", error]`，Runtime 在任务顺序变化、结果流式到达或同类返回值很多时，会遇到什么问题？

随后提供固定的数据类接口，由学习者一次只实现一个函数。不要先考察数据类语法，把注意力留给“身份对应与部分失败”。

## 8. 下一课验收与停止条件

- 能解释为什么结果必须保留 `call_id`。
- 能让单个工具失败不导致整个批次丢失成功结果。
- 能通过测试确认成功与失败结果分别属于正确调用。
- 若连续两次需要 C/D 级提示，缩小到单个工具包装函数，不继续增加并发上限或重试。
- 编码主练习完成后，可只听一个 Agent 主题：工具依赖图与并发调度；不要求当日实现。
